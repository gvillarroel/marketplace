## Resumen ejecutivo

Agent Harbor cumple el alcance funcional revisado para mantener y observar un
equipo de agentes en Pi, GitHub Copilot CLI y OpenCode. Las superficies de
equipo permiten identificar quién está disponible, quién está trabajando, el
estado de cada ejecución, su tarea pública redactada, el modelo y razonamiento
observables, tokens, coste, capacidad y ownership. Los controles deterministas
evitan llamadas al modelo cuando no son necesarias.

La revisión manual final incluyó un subagente crítico independiente. Su primera
pasada encontró cuatro brechas reproducibles; todas fueron corregidas y la
reauditoría devolvió `CLEAN`.

El alcance publicado en este corte reúne fuentes, artefactos compilados,
runtime empaquetado, documentación, pruebas y evidencias visuales. El estado
local generado durante la revisión manual bajo `.pi/agents` no forma parte del
producto ni de la publicación.

## Capacidades disponibles

- Roster persistente con managers, miembros fijos, bundled, personales,
  contractors y utility agents.
- Vistas `/team` compactas, filtrables y paginadas para roster, actividad,
  historial, diagnóstico y detalle exacto por run.
- Estados operativos visibles: ready/idle, starting, working, waiting, cleaning,
  completed, failed, cancelled, bench, stale, conflict y unavailable.
- Team lead con capacidad visible, máximo de seis delegaciones secuenciales y
  admisión fail-closed cuando no existe capacidad segura.
- Actividad persistente compartida entre Pi y Copilot con ownership,
  heartbeat, PID/ruta del proceso propietario y recuperación explícita.
- Telemetría nativa cuando el host la entrega: modelo, reasoning/thinking,
  input/output/reasoning/cache/total tokens y coste.
- `/bench`, `/join`, `/retire`, `/team` y catálogo paginado como rutas
  deterministas de 0 tokens.
- `/contract` mantiene exactamente un child desechable.
- Custom tools definidos en código común, con schemas cerrados, principal
  autenticado y registro dinámico sólo para el agente que las necesita.
- Sin servidor, registro ni transporte MCP.

## Correcciones finales relevantes

- Normalización decimal del coste de Pi para evitar artefactos IEEE-754 como
  `$0.000049999999999999996` sin convertir costes pequeños positivos en cero.
- Identidad ordinaria de modelo Pi conservada completa en el reporte terminal.
- Mensajes de actividad Copilot abreviados para evitar truncado visual.
- Abort nativo Copilot idempotente cuando ownership loss y timeout coinciden.
- Paginación real del catálogo con rutas next/previous y rechazo estricto de
  páginas inválidas o duplicadas.
- Ejemplo de ayuda OpenCode corregido a una ruta ejecutable:
  `/team status:bench page:2`.
- Ayuda Pi y Copilot actualizada con `[--page N]`.

## Riesgos y limitaciones conocidas

- Este corte se entrega mediante una rama y un PR; no debe confundirse el
  estado funcional verde con una release terminada hasta completar revisión,
  CI y merge.
- Durante un turno activo, Copilot 1.0.73 pausa comandos SDK: el progreso se
  publica automáticamente; `Esc` interrumpe y `/team` se consulta después del
  asentamiento.
- La telemetría de trabajo perteneciente a otro proceso no se revela; se
  muestran el estado compartido y la ruta exacta del owner para actuar allí.
- OpenCode no ofrece una operación host atómica compare-and-stop. Agent Harbor
  mitiga la carrera residual con verificaciones de engine, proyecto,
  generación, owner y turno, y falla cerrado ante incertidumbre.
- Los cambios de aliases dinámicos en OpenCode y Copilot pueden requerir reload
  para que el autocomplete refleje join, bench o retire.
- Las transcripciones manuales viven bajo `work/` y son evidencia local
  ignorada por Git; las conclusiones durables están resumidas en `docs/`.
